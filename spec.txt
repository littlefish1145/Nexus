# Nexus 改进方案 Spec

> 本文档不涉及代码改动，仅作为思路与设计参考。读者：项目维护者、贡献者。
> 范围限定：`/nexus` 目录下的 Go 实现。Web 管理控制台（#20）从本方案中移除。

---

## 1. 背景与现状评估

Nexus 是一款将"对象存储 + 加密微服务 + 存算一体管线 + 向量检索"融合在一个二进制里的系统。从代码与 README 来看，它已经做到了一件不容易的事：把 AWS S3 兼容网关、信封加密微服务、声明式管线、HNSW/IVF-PQ 索引、IAM/STS 体系装进一个 Go 进程里，并且提供分布式拆分能力。

但"功能完整"和"生产可用"之间还有显著距离。读完源码（`internal/`、`cmd/`、`proto/`、`policy/`）后，我的判断是：**加密与 IAM 的设计是亮点，但存储层、可观测性、生态兼容性、集群一致性是明显的短板**。以下 20 项改进按"对生态兼容性和生产可用性影响最大"排序，但写作时按"主题聚合"组织，便于通读。

需要先澄清的几点前提：

1. 改进方案**默认不破坏现有 API 与 protobuf 协议**；新功能以"新增子命令/新增 gRPC 方法/新增 HTTP 头"方式叠加。
2. 改进**默认保留 BoltDB 作为元数据存储**，仅在确实需要时讨论后端替换。
3. 分布式模式已经引入 Consul SDK，但**目前没有看到基于 Raft/etcd 的元数据共识实现**，这是一个未完成的承诺，需要明确补齐路径。
4. 文档较薄、测试不均是工程债，本方案会附带提及，但不会作为主线工作项。

---

## 2. 改进目标

| 维度 | 当前状态 | 改进后目标 |
|---|---|---|
| S3 生态兼容 | 缺 SSE-C、缺事件通知 | 与 awscli/MinIO SDK/boto3 默认行为完全对齐 |
| 数据安全 | 信封加密已落地，审计闭环不全 | 审计可聚合、可签名、可导出到 SIEM |
| 集群能力 | 节点间无共识 | 强一致元数据 + leader 选举 + 故障自愈 |
| 可观测性 | 手写 8 行 metrics 文本 | Prometheus 指标体系 + 分布式 tracing |
| 存储层 | 仅本地文件后端 | 冷存后端可插拔 + 纠删码可启用 |
| 向量检索 | 内存索引 | 大规模下可走 mmap / 磁盘索引 |
| 运维体验 | 缺备份恢复演练、缺健康检查 | 备份/恢复/告警链路完整 |
| 工程 | CLI 文本输出、无 SDK | 结构化输出 + 至少 2 个语言 SDK |

## 3. 详细改进项

### 3.1 SSE-C：客户提供密钥的服务端加密

**问题诊断**

S3 协议里 SSE-C 是一类一等公民：客户端生成一个 256-bit/128-bit 的密钥，每次请求把密钥（通过 HTTPS）+ 密钥的 SHA-256 哈希（`x-amz-server-side-encryption-customer-key` 和 `x-amz-server-side-encryption-customer-key-MD5`）发上来，服务端**绝不持久化**这个密钥。当前 `internal/gateway/s3.go` 在请求头白名单里**没有** `x-amz-server-side-encryption-customer-*` 这一组（搜了一遍只看到 `x-amz-server-side-encryption`），所以 awscli 默认走 SSE-C 时直接被网关拒掉。

**设计思路**

1. 在网关解析 `x-amz-server-side-encryption-customer-algorithm`（目前 S3 只允许 `AES256`），校验 `x-amz-server-side-encryption-customer-key-MD5` 与 base64 解码后的密钥的 MD5 一致。校验失败直接 400。
2. **不让密钥进任何持久化路径**——密钥只在请求作用域（`context.Context`）里存在。`EncryptionCoordinator` 增加 `EncryptWithClientKey(ctx, plaintext, clientKey)` 入口，密钥出 handler 即销毁。
3. 元数据里**只**记录密钥的 SHA-256 哈希和算法，作为审计与检索时的客户端密钥校验依据。
4. `GetObject` / `HeadObject` 路径要校验请求里带不带对应的 customer key，不带就 403，与 S3 行为一致。
5. DEK 的产生方式调整：正常路径下还是信封加密（每个对象一个 DEK，DEK 用 ECIES 长期密钥加密存到 KeyStore），但**数据本身用 DEK XOR/HKDF 派生出的子密钥 + 客户端密钥**混合后再 AES-GCM 加密——这一点需要密码学同事 review，但一个更简单的方案是：**客户端密钥**直接作为 DEK 使用，跳过信封加密，代价是无法去重。
6. 审计日志增加 `sse_c_used=true` 标志位，便于合规审计。

**取舍**

- 方案 A（混合加密）保留信封加密的去重能力，但实现复杂。
- 方案 B（直接以 client key 为 DEK）实现简单，但 `enable_dedup: true` 的语义必须重新审视——S3 自己也禁止对 SSE-C 对象去重。
- 建议先实现方案 B，在配置里禁止 SSE-C 路径下启用 dedup；后期再视需求加方案 A。

**风险**

- 客户端密钥泄露即数据泄露，无可恢复——必须配合强审计与告警。
- 误把客户密钥写日志：需要在 logger 增加 `redact` 列表，单元测试覆盖。

### 3.2 事件通知机制

**问题诊断**

S3 在对象级别提供 `s3:ObjectCreated:*`、`s3:ObjectRemoved:*` 等事件，可以通过 SNS/SQS/EventBridge/Lambda 触发下游。Nexus 现在没有事件出口，意味着上层业务要么轮询（浪费），要么用 webhook 自己实现（容易漏）。

**设计思路**

1. 在 metadata 层加一个**事件总线**（in-process 即可，初期无需外部 MQ）：
   - 触发点：`PutObject`、`CompleteMultipartUpload`、`DeleteObject`、`DeleteObjects`、`CopyObject`。
   - 事件载荷：`bucket`、`key`、`versionId`、`etag`、`size`、`eventName`、`timestamp`、`requesterIAMArn`、`sourceIP`。
2. 订阅模型：每个桶可以有 N 条规则（类似 S3 EventNotification），规则结构：
   ```yaml
   event: "s3:ObjectCreated:Put"
   prefix: "uploads/"
   suffix: ".jpg"
   destination:
     type: "webhook" | "kafka" | "nats" | "amqp"
     url: "https://..."
     signing: { algorithm: "hmac-sha256", secret: "..." }
   ```
3. 投递语义：**至少一次**。失败重试指数退避，失败 N 次进死信队列（落到 `data/deadletter/`）。
4. webhook 必须签名（HMAC-SHA256，header `X-Nexus-Signature`）并支持 `X-Nexus-Delivery` 唯一 ID 去重。
5. 接收方需要返回 2xx 才算成功；4xx 标记为可重试错误（仅对 5xx 与网络错误重试）。
6. 性能：事件投递**异步**，不影响主链路 P99。批量写入一个 ring buffer + worker pool。

**取舍**

- 内部事件总线足够 MVP；KAFKA/NATS 适配器可后置（参考 `internal/replication` 的 pluggable target 模式）。
- 事件顺序：S3 不保证跨 key 顺序，但**同一 key 强顺序**——需要按 key 分片到固定 worker。

**风险**

- 死信队列无监控会导致事件静默丢失，必须加指标 `nexus_deadletter_total` 和 `nexus_event_delivery_failed_total`。
- webhook 超时配置不当会拖垮 worker pool；建议默认 5s、bounded concurrency。

### 3.3 全文检索（BM25 倒排索引）

**问题诊断**

`internal/vector/` 只做向量检索，无法支持"包含某个关键词"或"标题中包含 X"的查询。混合检索（BM25 + 向量）已是行业标配（Elasticsearch、OpenSearch、pgvector hybrid search、Tigris 等）。

**设计思路**

1. 在 metadata 之外新增 `internal/fts/` 子系统，存储**倒排索引**。
2. 倒排索引结构：
   - dictionary：term → posting list（BoltDB bucket `fts_terms`）
   - posting list：`<docID, termFreq, [fieldPositions]>` 编码为 varint + delta
   - docID 即 `bucket#key#versionId` 的哈希
3. 分词：
   - 文本类（`text/*`、`application/json`）走分词；图片/视频走 OCR/ASR 衍生对象。
   - 默认 `unicode-segmentation` + 停用词表 + lowercase + porter stemmer（多语言可后置，先英文）。
4. 索引构建：
   - 管线在 `text_extract` 阶段产出 `tokens.bin`，`fts` 引擎消费后追加到倒排索引。
   - 增量构建：segment 化（每 1024 文档一个 segment），后台 merge（类似 LSM）。
5. 查询 API：新增 `/admin/fts/search?bucket=...&q=...&topK=...`，返回 `[{id, score, snippet, highlight}]`。
6. 与向量检索的协同：先在 metadata 层加 `hybrid_search` 入口，融合 BM25 + 向量分数（可配置 RRF 权重）。

**取舍**

- LSM 风格 segment 写入实现简单，但 merge 期间读放大——可接受。
- 跨 segment 评分需要在 query 时合并 posting list，内存峰值要测。

**风险**

- 中文分词不准确影响召回；MVP 可以只做英文 + 二元切分兜底。
- 倒排索引膨胀：需要定期 compaction 与磁盘配额。

### 3.4 断点续传上传（resumable / session-based upload）

**问题诊断**

`internal/gateway/multipart.go` 走的是经典 S3 multipart upload 协议（`CreateMultipartUpload` → `UploadPart` → `CompleteMultipartUpload`），但断线重传场景下：
- `UploadPart` 失败重试时，client 必须**重传整个 part**，因为协议不带 part checksum 协商。
- 服务端没有为 part 提供 **streaming dedup** 能力。
- `uploadId` 续期 / 列出已上传 parts 的便利接口缺失。

**设计思路**

1. 借鉴 tus.io 协议，**可选**地启用 resumable upload（通过 `X-Nexus-Resumable: 1` 头协商）：
   - `POST /{bucket}/{key}?resumable` 创建 upload session，返回 `Location` + `Upload-Offset: 0`。
   - `PATCH /{bucket}/{key}?uploadId=...` 上传 chunk，body 是 raw bytes，header `Upload-Offset` 校验；返回 `Upload-Offset: <new>`。
   - `HEAD /{bucket}/{key}?uploadId=...` 查询当前 offset（断线恢复关键）。
   - 服务端**必须**保证 `Upload-Offset` 单调递增，且不缓冲整个文件。
2. 服务端实现：每个 session 写一个 `data/uploads/{uploadId}.tmp` 文件，PATCH 追加写；HEAD 读文件大小即 offset。
3. 与 multipart 的关系：resumable session 完成时（client 发 `X-Nexus-Finalize: 1`），把 `.tmp` 重命名为最终对象，并触发管线与加密。
4. 清理：session 超过 `expires_in`（默认 24h）由后台清理任务删除 `.tmp`。
5. 指标：`nexus_resumable_session_active`、`nexus_resumable_session_completed`。

**取舍**

- 协议自定义而非复用 S3 multipart，因为 S3 协议本身就不支持断点续传语义。
- 可以做同时支持：默认走 S3 multipart，`X-Nexus-Resumable` 头开启时切到 tus 风格。

**风险**

- `.tmp` 残留必须主动清理，否则磁盘泄漏。
- 续传校验：必须按 SHA256/CRC32C 校验每段，避免"上传一半客户端崩溃但服务端以为成功"。

### 3.5 存储后端可插拔 + 纠删码

**问题诊断**

`internal/storage/store.go` 只有 `FileBackend`（本地文件系统）。Archive 层配置 `archive_path: ""` 也是本地盘。这导致：
- 没有跨节点副本，单盘故障 = 数据丢失（除非依赖外部备份）。
- 没有低成本冷存后端对接 S3 Glacier / Azure Archive / 阿里云 OSS Archive。
- 副本策略仅靠 `internal/replication` 异步复制，没有同步强一致副本。

**设计思路**

1. 后端抽象：`BackendStorage` 接口已经存在，扩展为：
   ```go
   type BackendStorage interface {
       io.Closer
       Name() string
       Put(ctx, path, data, size) error
       Get(ctx, path) (io.ReadCloser, error)
       Delete(ctx, path) error
       Exists(ctx, path) (bool, error)
       Size(ctx, path) (int64, error)
       List(ctx, prefix) ([]string, error)
       // 新增
       PutReader(ctx, path, io.Reader) (etag string, err error)
       GetRange(ctx, path, start, length) (io.ReadCloser, error)
       AtomicRename(ctx, oldPath, newPath) error
   }
   ```
2. 新增后端：
   - `S3Backend`：把数据当对象存到 AWS S3 / MinIO / 任何 S3 兼容服务。
   - `AzureBlobBackend`：对接 Azure Blob Storage。
   - `SftpBackend`：通过 SFTP 写到远程归档。
3. **纠删码**（Reed-Solomon）作为可选项：
   - 在 `BackendStorage` 之上包一层 `ErasureCodedBackend`：`Put` 时把数据切成 k 个数据片 + m 个校验片，分布到不同后端 / 不同节点 / 不同 AZ。
   - `Get` 时任取 k 片即可还原。
   - 推荐 k=4, m=2（容忍 2 节点同时故障）。
4. 桶级别配置 `storage_class: "hot" | "warm" | "cold" | "archive"`，每个 class 绑定一个 backend。
5. **冷热分层**已经有 `internal/tiering/manager.go`，需要补：
   - Cold/Archive 后端的 `Put/Get` 必须支持**大对象流式**（multipart 上传到 S3 backend）。
   - 异步迁移任务用 `internal/scheduler` 触发。
6. 副本策略：与现有的 `internal/replication` 整合，配置 `replication_factor: 3` + `replica_strategy: "sync" | "async"`。

**取舍**

- Reed-Solomon 实现建议直接用 `github.com/klauspost/reedsolomon`，成熟度高。
- 同步副本对延迟敏感：sync 副本在跨 AZ 部署下 P99 会涨 1 个 RTT，要给 `replica_strategy: "async"` 留好默认。

**风险**

- 纠删码在数据恢复期间磁盘 IO 与 CPU 飙升，需要限制后台重建并发。
- S3 backend 的 part size 与 multipart threshold 要测，默认 8MB 比较稳妥。

### 3.6 向量索引持久化（mmap / 磁盘索引）

**问题诊断**

`internal/vector/engine.go` 的 HNSW/IVF-PQ 都是 in-memory。`vector.hot_index_size: 10GB` 的配置在单节点上意味着最多 10GB 浮点向量（768 维 × 4 字节 = 3KB/向量，约 350 万条）。超过这个量：
- 进程重启 = 重建索引（线性扫描全对象 → 调 embedding → 重建图），动辄小时级。
- 单机内存成为瓶颈。

**设计思路**

1. 引入 **HNSW mmap 索引**：
   - 把 HNSW 图序列化到 `data/vector/{bucket}.hnsw`，节点和边用 memory-mapped 访问。
   - 借鉴 `hnswlib-go` 的磁盘布局：每层一个文件，分块 mmap。
2. 引入 **DiskANN** 风格（可选）：微软的 DiskANN 用 Vamana 图 + 量化 + PQ 压缩，专门为 SSD 设计，比 HNSW 在 1 亿+ 向量规模下内存占用低 10x。
3. 量化层：
   - `scalar_quantization`: float32 → int8，单边 4x 压缩，召回率 99%+。
   - `product_quantization`: 把向量切成 8 段，每段聚类到 256 个 centroid，压缩比 32x，召回率 95% 左右。
   - 在 `internal/vector/embedding.go` 增加 `Quantize(v) []byte` 与 `Dequantize(b)`。
4. 索引分片：
   - 桶级别独立索引文件，避免单索引文件过大。
   - 跨桶查询走 broker 模式：先按桶过滤再合并 topK。
5. 重建流程：
   - 后台 goroutine 监听 `vector.rebuild` CLI 命令，串行写新索引文件，原子 rename 替换。
   - 旧索引在替换后保留 N 分钟作为 fallback。

**取舍**

- DiskANN 引入复杂度高，建议先做 mmap HNSW + SQ/PQ 量化，DiskANN 作为 2.0 路线。
- 量化会损失召回率，metric 必须用 cosine（IP 量化误差大）。

**风险**

- mmap 在 Windows 上的行为与 Linux 不同，跨平台要测。
- 索引损坏 = 数据丢失（向量维度的内容不可重建），必须有索引 SHA256 校验与重建工具。

### 3.7 内容去重（CAS / Content-Addressable Storage）

**问题诊断**

`config.yaml` 里 `encryption.enable_dedup: true` 但搜遍 `internal/` 没有看到 content-hash 去重逻辑。Nexus 当前每个对象都是独立存储路径，没有 CAS 语义。

**设计思路**

1. 客户端上传时（PUT 与 multipart）服务端计算 SHA-256 增量摘要（incremental hashing，避免内存里放整文件）。
2. 存储路径由 `content_hash` 派生：`data/cas/{shard}/{hash[:2]}/{hash}`。
3. 元数据里记录 `etag = hash`，并维护 `refcount`：
   - 多个对象相同内容时物理只存一份。
   - `DeleteObject` 时 refcount -1，归零才物理删除。
4. 与加密的协调：
   - 因为信封加密的 DEK 是**每对象独立**的，去重只能发生在"明文相同 + 加密密钥相同"时。
   - 简化方案：仅当**未加密**或**SSE-C 模式**下去重。SSE-S3 不去重（与 S3 行为一致）。
5. 配额：去重后 `BucketInfo.TotalSize` 计**逻辑大小**还是**物理大小**？建议双指标：
   - `total_size_logical` 用于计费/quota。
   - `total_size_physical` 用于磁盘规划。

**取舍**

- 与加密冲突的取舍：先去重，再加密，可以让 `enable_dedup` 仅在 `kms_type: "none"` 或 `sse_c_only: true` 时生效。
- 与管线的协调：管线的衍生对象（thumbnail、压缩版）天然适合 CAS，因为同一原图每次生成的 thumbnail 哈希相同。

**风险**

- refcount 维护出错 = 数据丢失或泄漏。必须用事务（BoltDB 的 `Update` 事务）保证 hash 插入与 refcount 增减的原子性。
- 热点 hash：热门文件 hash 相同导致单 shard IO 集中，需要 shard 散列到 N 目录。

### 3.8 集群共识（Raft / etcd）

**问题诊断**

`config.yaml` 里有 `cluster_peers: []`，go.sum 里有 `consul/sdk`，但代码里没有看到元数据层的 leader 选举。现状是：**多节点部署时元数据各自维护，最终一致性靠"复制"，没有强一致保证**。这意味着 IAM 策略生效、桶 ACL 变更、对象版本号分配都有竞态。

**设计思路**

1. 引入 Raft 共识库（建议 `github.com/hashicorp/raft`）。
2. 元数据操作（IAM、桶元数据、对象版本号、引用计数）都走 Raft log：
   - 网关层接到写请求后，调用 `RaftApply(ctx, op)` 阻塞直到 log 提交。
   - BoltDB 作为 Raft 的 FSM（finite state machine）snapshot。
3. 节点角色：
   - **leader**：处理所有写。
   - **follower**：处理读（线性化读需要 `VerifyLeader`+`ReadIndex`）。
   - **candidate**：选举中。
4. 启动流程：
   - 单节点直接 bootstrap。
   - 多节点通过 `cluster_peers` 配置种子，raft 自动 join。
5. 数据分片（可选）：Raft 适合控制面（小数据量），数据面（对象内容）走 `replication`。
6. **Consul 的角色**：仅做服务发现 + 健康检查，不参与元数据共识——把这条线明确下来。

**取舍**

- Raft vs etcd：自维护 raft 嵌入进程零依赖，但需要自己处理 snapshot/恢复；etcd 部署运维重。
- 建议自维护 raft，控制面走 raft，数据面走对象复制（CRDT-lite 风格），桶级 metadata 也走 raft。

**风险**

- Raft 性能：批量写 + pipeline apply 优化；元数据层 1k QPS 写可以扛，10k+ 需要分片 raft。
- 脑裂防护：必须有 pre-vote + check quorum，避免网络分区时双 leader 写冲突。
- 升级路径：从单节点平滑升级到 3 节点集群，需要"learner → voter"过渡。

### 3.9 可观测性（Prometheus + OpenTelemetry）

**问题诊断**

`/admin/metrics` 端点手写 8 行文本（`internal/gateway/auth.go:905`、`admin_auth.go:629`），只暴露 `nexus_up` 一个布尔。运维需要的核心指标全缺：
- 请求 QPS / P50/P95/P99 延迟
- 各微服务的 RPC 错误率
- 存储 IO 吞吐
- 向量索引大小、查询延迟
- 加密操作的端到端耗时

**设计思路**

1. **指标**（Prometheus）：
   - 接入 `github.com/prometheus/client_golang`。
   - 中间件自动埋点：HTTP 请求按 method/path/status 聚合，histogram 记录延迟。
   - gRPC 客户端/服务端拦截器记录 `rpc_duration_seconds`。
   - 自定义指标：
     - `nexus_object_put_total{bucket,status}`
     - `nexus_object_get_bytes_total{bucket}`
     - `nexus_encrypt_duration_seconds{service}`
     - `nexus_vector_index_size_vectors`
     - `nexus_vector_search_duration_seconds{index_type}`
     - `nexus_storage_io_bytes_total{op,backend}`
     - `nexus_replication_lag_seconds{rule}`
     - `nexus_event_delivery_total{status}`
     - `nexus_iam_policy_eval_duration_seconds`
2. **Tracing**（OpenTelemetry）：
   - 选 OTLP gRPC exporter（兼容 Jaeger / Tempo / 阿里云 ARMS）。
   - HTTP / gRPC 入口自动生成 trace span；加密微服务之间传播 trace context（`traceparent` header）。
   - 关键 span：`gateway.put_object` → `coordinator.encrypt` → `keygen.generate` → `encrypt.encrypt` → `storage.put` → `pipeline.execute` → `vector.index`。
3. **日志**：保留现有 `internal/logger/logger.go`，但用 `slog` 替换自写 logger（Go 1.21+ 标准库自带），统一 JSON 格式并带 trace_id / span_id。
4. **健康检查**：
   - `/healthz`：进程存活。
   - `/readyz`：依赖项（DB、gRPC 服务）全部就绪。
   - `/livez`：与 healthz 类似但更轻量（用于 k8s liveness probe）。
5. **告警样例**（写到 `docs/runbook.md`）：
   - 加密 RPC 错误率 > 1% 持续 5min
   - replication_lag > 60s
   - 磁盘使用率 > 85%
   - vector_search P99 > 1s

**取舍**

- Prometheus + OTLP 两条线分开，避免指标打满 tracing 带宽。
- metric cardinality 注意：`bucket` 标签数量大，可能导致 TSDB 爆炸，需要限制 topN 桶打 tag。

**风险**

- 高 QPS 场景下 metrics 自身开销：histogram bucket 数量要克制（默认 5-7 个 bucket 即可）。
- OTLP exporter 的 batching 配置不当会造成 trace 丢失。

### 3.10 备份与恢复（增量 + 演练）

**问题诊断**

`internal/backup/backup.go` 看起来是 bbolt 全量拷贝 + 文件级 rsync 思路。问题：
- 没有增量备份，每次都是全量 → 备份窗口大。
- 没有恢复演练工具。
- 没有跨节点一致性保证（备份时元数据还在写怎么办？）。
- 没有异地加密备份。

**设计思路**

1. **BoltDB 端**：用 bbolt 的 `Tx.WriteTo` 拿到一致性快照（bbolt 是单写事务模型，备份期间 read-only view 即可）。
2. **数据文件端**：用 `data/{bucket}/` 下的对象文件 + `.meta` sidecar。
3. **增量备份**：
   - 维护一个 `last_backup_lsn`（log sequence number，元数据层递增）。
   - 备份时只导出 `lsn > last_backup_lsn` 的对象 + 元数据变更。
   - 格式：tar 增量包 + manifest.json（记录 lsn 范围、checksum、对象计数）。
4. **异地加密**：
   - 备份包用 AES-256-GCM 加密，密钥从 KMS 取，不硬编码。
   - 备份上传到 S3 / OSS 远端 bucket，启用服务端加密。
5. **恢复**：
   - `nexusctl backup restore --from <path>` 把全量 + 增量依次 apply 到目标 data 目录。
   - 恢复前必须 `nexusctl backup verify --from <path>` 校验完整性。
6. **演练**：
   - `nexusctl backup drill` 启动一个临时 Nexus 实例，喂入备份数据，验证可启动、可读、可查询。
   - 演练结果以指标形式上报：`nexus_backup_drill_success_total`。

**取舍**

- 增量备份实现复杂度高，MVP 可以先做"按桶备份"（每个桶一个增量包），简化 lsn 模型。

**风险**

- 备份与生产 IO 抢带宽：限速 + 业务低峰期触发。
- 恢复时的权限：恢复时管理员必须有最高权限，且操作要二次确认。

### 3.11 IAM 表达力增强

**问题诊断**

`internal/iam/policy.go` 看起来是简单的字符串匹配式策略评估（与 AWS IAM 早期版本类似）。缺：
- 条件键（`aws:RequestTag`、`s3:Prefix`、`aws:CurrentTime` 范围等）
- ABAC（基于 tag 的访问控制）
- Permission Boundary（用户级权限上限）
- Service Control Policy（org 级护栏）

**设计思路**

1. **条件键**：
   - 策略 JSON 里支持 `"Condition": { "StringEquals": { "s3:prefix": "uploads/" } }`。
   - 评估器在 `policy.go` 增加 condition evaluator：先看操作匹配，再看资源匹配，最后看 condition。
   - 内置条件操作符：`StringEquals`、`StringLike`、`NumericEquals`、`DateGreaterThan`、`IpAddress`、`Bool`、`ArnEquals`、`Null`。
2. **ABAC**：
   - 桶/对象支持 tag，用户/角色支持 session tag。
   - 策略里写 `"aws:ResourceTag/Project": "${aws:PrincipalTag/Project}"` 实现 tag-based 授权。
3. **Permission Boundary**：
   - 给用户/角色绑一个边界策略，实际权限 = 身份策略 ∩ 边界策略。
4. **SCP**（组织级）：
   - 节点级配置，约束所有 IAM 决策。
5. **策略模拟器**：`nexusctl iam simulate --action s3:PutObject --resource arn:...` 返回 Allow/Deny + 命中语句。

**取舍**

- 条件键实现量大，建议按需增量：先 `StringEquals` / `StringLike` / `NumericEquals` 三种，覆盖 80% 场景。

**风险**

- condition 评估器 bug = 安全绕过，必须有 fuzz 测试（用 `go-fuzz` 或 property-based test）。

### 3.12 CLI 体验：结构化输出 + 补全脚本

**问题诊断**

`cmd/nexusctl/*.go` 全是表格化 stdout 输出，没有 `--json` / `--yaml` 选项。外部工具（CI、Ansible、监控）无法直接消费。

**设计思路**

1. 全局 `--output=json|yaml|text` 标志：
   - text：现有表格。
   - json/yaml：序列化到 stdout，方便 `jq` / `yq` 处理。
2. 子命令补全脚本：
   - `nexusctl completion bash > /etc/bash_completion.d/nexusctl`
   - 同时支持 zsh / fish / powershell。
3. `--query` JMESPath 过滤：例如 `nexusctl bucket list --output json --query '[?region==`us-east-1`].name'`。
4. 配置文件：支持 `~/.nexusctl.yaml` 存 access key / secret / endpoint，免每次输入。
5. 错误输出统一为 RFC 7807 风格 JSON，便于机器解析。

**取舍**

- JMESPath 库选 `github.com/jmespath/go-jmespath`。
- 配置加密：本地密码用 OS keyring，不存明文。

**风险**

- 配置文件泄露 = S3 凭据泄露。文档必须强调权限 0600。

### 3.13 官方 SDK

**问题诊断**

没有 Go / Python / Java 客户端 SDK。外部业务方只能手搓 HTTP + SigV4，集成成本高，且容易在签名/时间戳/分块校验上出错。

**设计思路**

1. **Go SDK**（`nexus-sdk-go`）：
   - 上游仓库，引用 `github.com/aws/aws-sdk-go-v2` 的 S3 client interface，**让用户无感替换**：
     ```go
     import "github.com/nexus/sdk-go/s3"
     // 与 s3.NewFromConfig 用法一致
     ```
   - 这意味着：endpoint 换成 `http://nexus.local:8080`，其余代码不动。
2. **Python SDK**（`nexus-sdk-python`）：
   - 复用 `boto3` 客户端：
     ```python
     import boto3
     s3 = boto3.client('s3', endpoint_url='http://nexus.local:8080', ...)
     ```
   - 仅在需要 Nexus 特有功能（管线触发、向量检索、事件订阅）时暴露扩展方法。
3. **Java SDK**：参考 `aws-sdk-java-v2` 的 S3 client SPI 模式，做成可选 plugin。
4. **SDK 一致性测试套件**：
   - 用 [s3-test-suite](https://github.com/ceph/s3-test-suite) 或自维护的 conformance test，覆盖 S3 协议核心 API。

**取舍**

- Go SDK 是必选；Python/Java 可以"通过 boto3/SigV4 复用"作为 MVP，正式 SDK 排期靠后。

**风险**

- SDK 一旦发布就是契约，必须有版本兼容策略（不要轻易 break 旧签名）。

### 3.14 加密审计聚合 + 签名链

**问题诊断**

7 个微服务各自写自己的 audit log（`crypto_services.audit_size: 10000` 提示是 ring buffer），但没有聚合层。运营方需要在 7 个文件里 grep 关联一次加密操作（按 token id / object key 串起来）。

**设计思路**

1. 统一事件格式：
   ```json
   {
     "event_id": "uuid",
     "trace_id": "otel trace id",
     "span_id": "otel span id",
     "timestamp": "RFC3339 nano",
     "service": "KeyGenService",
     "operation": "GenerateDataKey",
     "subject": { "user": "...", "iam_arn": "..." },
     "object": { "bucket": "...", "key": "..." },
     "token_id": "...",
     "result": "success|failure",
     "error": "..."
   }
   ```
2. 聚合器：每服务在写本地 ring buffer 的同时**异步**推送到 audit aggregator（gRPC stream），aggregator 落到 `data/audit/` 目录的 hourly 文件里。
3. 签名链：每小时文件整体 SHA-256 摘要 + Ed25519 签名（用专门的 audit signing key，**与加密密钥完全隔离**），签名存到 `data/audit/_signatures/{hour}.sig`。
4. 导出：aggregator 支持 syslog / Kafka / S3 投递，转给 SIEM。
5. 不可篡改：日志文件 append-only，签名链 + WORM 存储（写入后无法修改）。

**取舍**

- 签名链的密钥管理：用 HSM 最佳，但 MVP 可以用 KMS 加密存储的 Ed25519 私钥。

**风险**

- audit 写入失败 = 合规事故，需要监控 + 自动告警。

### 3.15 性能：加密粒度与小对象优化

**问题诊断**

信封加密的设计是"每对象一个 DEK"。对 1KB 的小对象：
- DEK 32 字节 + ECIES 加密后约 100 字节 + ECDH 加密 + 元数据 ≈ 200 字节固定开销。
- 小对象占比 > 30% 时，**密钥元数据** 占用可观磁盘和 IO。

**设计思路**

1. **对象组共享 DEK**（可选）：
   - 同一桶/同一 prefix 下、同一时间窗口内、同一用户的对象共享一个 DEK。
   - DEK 加密对象范围在元数据里记录（`dek_id` + `dek_generation`）。
2. **流式 CTR + per-chunk nonce**：
   - 大对象（> 1MB）改用 AES-CTR + HMAC-SHA256（encrypt-then-MAC），每 64KB 块一个 nonce。
   - 优势：可并行加密/解密（多核利用），解密时损坏只影响单块。
3. **客户端加密选项**：
   - 文档明确建议"大对象走 SSE-S3，小对象走 SSE-C 由客户端管理"。

**取舍**

- 对象组共享 DEK 削弱了"逐对象撤销"能力（撤销一个 DEK = 撤销一组对象），需明确告知。
- AES-CTR 不像 GCM 那样 AEAD，需要自实现 MAC，bug 风险高。

**风险**

- 任何密码学实现改动都要有 test vector（如 NIST CAVP）与 RFC 文档对应。

### 3.16 KMS 集成（Vault Transit / AWS KMS）

**问题诊断**

`config.yaml` 里有 `kms_type: "vault"`、`vault_addr`、`vault_transit_key`、`aws_kms_key_id`、`aws_region`，但搜遍 `internal/bootstrap/crypto.go` 与 `internal/services/keygen_service/` 没看到对应 client 实现。配置项是**空头支票**。

**设计思路**

1. **统一 KMS 抽象**：
   ```go
   type KMSClient interface {
       GenerateDataKey(ctx, keyID, length) (plaintext, encrypted []byte, err error)
       DecryptDataKey(ctx, keyID, encrypted []byte) (plaintext []byte, err error)
       GetPublicKey(ctx, keyID) (pub []byte, err error)
   }
   ```
2. **LocalKMS**：当前实现，文件存 ECDSA 长期密钥。
3. **VaultTransitKMS**：
   - 用 Vault Transit 引擎的 `datakey` API：服务端生成 DEK 并返回明文+密文。
   - Nexus 自身不持有长期密钥，私钥由 Vault 保管。
4. **AWSKMS**：
   - `GenerateDataKey` / `Decrypt` API 直接对 AWS KMS。
   - 公钥用 `GetPublicKey`。
5. **密钥轮换**：
   - `nexusctl crypto rotate --key-id ...` 触发 KMS 侧轮换 + 元数据层指向新密钥版本。
   - 旧 DEK 用旧 key 版本加密的还能解（KMS 保留历史版本），后台任务逐步 rewrap。

**取舍**

- Vault Transit 是"信封加密外包"的成熟方案，推荐优先实现。

**风险**

- KMS 依赖网络：网关必须容错（KMS 短时不可用时降级为只读？直接拒绝写入？需要明确策略）。

### 3.17 配置 schema 强校验

**问题诊断**

`config.go`（`internal/config/config.go`）大概率是 `mapstructure` + 反射装配，启动时填错字段（如 `vector.dim: 1000` 但模型是 768 维）会在运行时崩溃，且错误信息晦涩。

**设计思路**

1. 引入 schema 定义：用 `cue` 或 `json-schema` 定义 config 形态。
2. 启动时 `config validate --file config.yaml` 显式校验，错误信息包含字段路径 + 期望值。
3. 类型约束：`vector.dim` 必须是 768 的倍数；`tiering.hot_max_size` 必须是带单位字符串且能解析。
4. 跨字段约束：`vector.dim` 与 `vector.embedding_model_path` 加载后实际维度不匹配 → 启动失败并提示。
5. **热加载**（可选）：监听 SIGHUP 重新加载 config，部分字段允许运行时变更（如 rate limit、log level），部分必须重启（如 storage backend、kms key）。

**取舍**

- CUE 比 json-schema 表达力更强，**推荐 CUE**。

**风险**

- schema 维护成本：每次加新配置都要同步更新 schema。

### 3.18 测试与质量保障

**问题诊断**

带 `*_test.go` 的目录有 `cache`、`config`、`metadata`、`ratelimit`、`replication`、`scheduler`。`gateway`、`iam`、`pipeline`、`vector`、`services` 关键路径测试稀疏。CI 没有看到（仓库里没有 `.github/workflows`）。

**设计思路**

1. **覆盖率门槛**：
   - 新代码 `go test -cover` 必须 ≥ 80%。
   - `internal/services/encrypt_service`、`decrypt_service`、`keygen_service`、`keyunwrap_service` 这种密码学代码必须 100% 覆盖。
2. **集成测试**：
   - 启动一个临时 Nexus 实例 + 临时 BoltDB，跑完整 PUT/GET/DELETE/管线/向量检索。
   - 加密微服务用 gRPC client 连真实进程（testcontainers 风格）。
3. **端到端 conformance**：
   - 跑 s3-test-suite 的核心 case（Get/Put/Delete/List/Multipart/Versioning/ACL）。
4. **Fuzz**：
   - `policy.go`、`crypto.go`、`sigv4.go` 必须有 fuzz target（`go test -fuzz`）。
5. **CI**：
   - GitHub Actions：`go test ./... -race -count=1`、`go vet`、`staticcheck`、`golangci-lint`。
   - 每次 PR 必须通过。
6. **性能基准**：
   - `go test -bench` 跑标准 benchmark（put/get throughput、向量检索 P99），结果存到 `docs/benchmarks/`。

**风险**

- 测试运行时间膨胀：integration test 用 t.Parallel + build tag 区分（`//go:build integration`）。

### 3.20 启动脚本与部署

**问题诊断**

`start_services.py` 是个便利脚本（仅本地开发用），没有 systemd unit / docker-compose / k8s manifest。生产部署需要用户自组装。

**设计思路**

1. **systemd**（`packaging/systemd/`）：
   - 每个微服务一个 unit：`nexus-encrypt.service` 等。
   - `nexus.target` 把所有 unit 拉起来。
2. **docker-compose**（`deploy/docker-compose.yml`）：
   - 7 个微服务 + 主进程，共享 network + volume。
   - healthcheck + depends_on。
3. **Helm chart**（`deploy/helm/nexus/`）：
   - Deployment + Service + ConfigMap + Secret + PVC。
   - Ingress + cert-manager 集成。
   - HPA（基于 Prometheus adapter 的 CPU/内存/自定义指标）。
4. **k8s operator**（可选，长期）：
   - CRD：`NexusCluster`，operator 负责调协节点状态、滚动升级、备份调度。

**取舍**

- MVP 给 docker-compose 即可；helm chart 排期靠后；operator 是 2.0 路线。

**风险**

- 镜像体积：Go 编译产物 30MB 量级，用 multi-stage build 后镜像 < 100MB。

| 风险类别 | 具体风险 | 缓解 |
|---|---|---|
| 协议兼容 | SSE-C、事件通知可能与 S3 细节差异 | 用 s3-test-suite 跑 conformance；先在 feature flag 后默认开启 |
| 密码学 | 加密实现改动可能引入后门 | NIST CAVP test vector + 密码学同事 review + 100% 覆盖 |
| 集群切换 | 单节点 → 集群的元数据迁移 | 提供 `nexusctl cluster migrate` 工具；Raft learner 模式过渡 |
| 性能回归 | 新指标/tracing 拖慢热路径 | 异步上报 + 限速 + benchmark 门禁 |
| 数据迁移 | SSE-C / 去重 / 加密粒度改造需要重写对象 | 提供 `nexusctl rewrap --bucket ...` 工具；按桶分批执行 |

## 6. 推荐落地路径

1. **阶段 1（生态兼容，最高 ROI）**：
   - SSE-C（3.1）
   - 事件通知（3.2）
   - CLI 结构化输出（3.12）
2. **阶段 2（生产可用）**：
   - Prometheus + OTel（3.9）

## 7. 度量指标

每个改进项落地后必须可度量：

- **功能**：`s3-test-suite` 通过率；自定义 conformance 用例数。
- **性能**：PUT/GET P99 延迟；加密 RPC P99；向量检索 P99。
- **稳定性**：MTTR（恢复时间）；备份恢复演练成功率。
- **可观测**：metric 数量（应 > 50 个核心指标）；trace 覆盖率（应有 > 90% 关键路径打点）。
- **安全**：fuzz 跑时长（≥ 1h 无崩溃）；密码学测试向量通过率（100%）。
- **开发者体验**：CLI `--output json` 覆盖子命令数；SDK 单元测试覆盖率（≥ 85%）。
